# PipeCAD lib/apps 架构重构实施计划书

> 版本：1.0  
> 日期：2026-04-05  
> 状态：待执行

## 1. 文档目的

本文档用于将 PipeCAD 的架构重构工作从“方向性方案”细化为“可执行任务计划”。

本计划聚焦以下四个目标：
1. 将代码根结构重构为 src/lib 与 src/apps。
2. 将 src/lib 统一收敛为一个对外暴露的架构静态库。
3. 将 src/apps 设计为支持多个 app 的业务目录，目前先承载一个 app。
4. 在架构层先行引入 C++20 Modules 与线程安全基础设施，为后续后台计算能力铺路。

本文档不直接替代架构设计文档，而是作为后续实施和验收的执行计划。

## 2. 总体原则

### 2.1 结构原则
- src/lib 是唯一架构层根目录。
- src/apps 是唯一业务与产品层根目录。
- src/lib 对外只暴露一个统一静态库目标。
- src/apps 下每个 app 独立装配、独立产物、统一链接 src/lib。

### 2.2 依赖原则
- lib 不允许反向依赖任意 app。
- apps 不允许直接依赖 OCCT、VSG、VTK 头文件。
- apps 通过 lib 的公开接口访问平台能力与运行时能力。
- C++20 Modules 先在 lib 内部推进，再向 apps 扩展。

### 2.3 线程安全原则
- 主线程持有全部可变文档状态、事务状态、场景状态和 Qt/QML 对象。
- 后台线程只允许在快照和只读接口上工作。
- apps 不自定义并发模型，统一使用 lib 提供的任务调度与结果回投机制。

## 3. 目标结构

### 3.1 src/lib 子域
1. base：基础类型、日志、信号、数学工具、基础并发原语。
2. platform：OCCT、VSG、VTK 封装与 façade。
3. runtime：Document、DependencyGraph、事务管理、命令管理、序列化、任务调度、快照。
4. framework：Application、AppContext、Workbench 宿主、SelectionManager、SceneManager、生命周期装配。

### 3.2 src/apps 子域
当前 app 建议命名为 pipecad，并采用如下子域：
1. model：业务模型。
2. engine：业务引擎与推导逻辑。
3. workbench：工作台实现。
4. ui：控制器、Qt Model、QuickItem、QML。
5. resources：图标与资源。
6. main.cpp：启动入口。

## 4. 里程碑总览

| 里程碑 | 名称 | 目标 | 输出物 |
|------|------|------|------|
| M0 | 规则冻结 | 冻结目录、命名、依赖和线程边界规则 | 计划文档、命名规则、依赖规则 |
| M1 | 构建重编组 | 在不大迁目录前先形成 lib/apps 逻辑结构 | 新的 CMake 拓扑、过渡别名 |
| M2 | 目录迁移 | 完成核心代码从旧目录到新目录的物理迁移 | src/lib、src/apps 目录成型 |
| M3 | lib 模块化 | 在 lib 内落地第一批 C++20 Modules | lib 模块接口单元、兼容头 |
| M4 | 并发基础设施 | 建立线程安全快照与任务调度基础设施 | 快照、任务队列、回投机制 |
| M5 | 异步重算落地 | 完成几何重算与高价值热点后台化 | 异步重算链路、回归测试 |
| M6 | 收口与模板化 | 清理兼容层，固化多 app 结构模板 | 最终目录、文档、测试入口 |

## 5. 任务编号规则

任务编号采用以下规则：
- M0-T01 表示里程碑 M0 下的第 1 个任务。
- 每个任务必须包含：目标、输入、改动范围、交付物、依赖、验收标准。
- 每个任务完成后，应同步更新状态表与日志。

## 6. 详细实施任务

### M0 规则冻结

#### M0-T01 冻结目录与目标命名规则
目标：确定 src/lib 与 src/apps 为唯一长期代码根目录，并冻结目标命名规则。

改动范围：
- docs
- 顶层 CMake 设计说明

交付物：
- 本计划书
- 目录规则说明
- target 命名规则说明

依赖：无。

验收标准：
- 明确架构库对外名称。
- 明确业务库与可执行名称规则。
- 明确未来新增 app 的扩展方式。

#### M0-T02 冻结 include 与 import 规则
目标：定义 lib 与 apps 的可允许依赖路径和禁止路径。

交付物：
- include/import 规则章节
- 第三方库访问约束

依赖：M0-T01。

验收标准：
- apps 不允许直接 include OCCT、VSG、VTK。
- lib::runtime 不允许反向依赖 lib::framework。
- 业务层不能跨 app 直接依赖。

#### M0-T03 冻结线程安全边界
目标：定义主线程对象、后台线程对象和快照边界。

交付物：
- 线程边界清单
- 主线程专属对象列表
- 后台线程允许行为列表

依赖：M0-T01。

验收标准：
- 命令管理、事务管理、Document 写操作、Scene 写操作的主线程所有权明确。
- 后台线程只允许消费只读快照。

### M1 构建重编组

#### M1-T01 设计新的 CMake 顶层装配拓扑
目标：将当前 src 目录下的目标组织方式重设计为 lib/apps 双根结构。

输入：
- 现有 src/CMakeLists.txt
- 现有各子目录 CMakeLists.txt

交付物：
- 新的目标依赖图
- 过渡阶段的 add_subdirectory 方案

依赖：M0 全部任务。

验收标准：
- 存在统一架构库目标。
- 存在单个 app 业务目标与可执行目标。

#### M1-T02 解除 app 对 RecomputeEngine 源文件级拼接
目标：消除当前 app 静态库直接吞并 RecomputeEngine.cpp 的结构性阻塞。

交付物：
- 新的重算链接方案
- 替代性的目标依赖或接口依赖设计

依赖：M1-T01。

验收标准：
- 不再存在源文件级跨目录拼接。
- 重算链路仍可正常构建。

#### M1-T03 设计过渡兼容层
目标：在不一次性改完所有 include 的前提下，提供平滑迁移路径。

交付物：
- 兼容 include 规则
- 旧路径到新路径的映射表

依赖：M1-T01。

验收标准：
- 能支持阶段性迁移。
- 不要求一次性替换所有 include。

### M2 目录迁移

#### M2-T01 建立 src/lib 目录骨架
目标：创建 src/lib/base、platform、runtime、framework 基础目录与命名空间规划。

交付物：
- src/lib 目录结构
- 对应 CMakeLists 入口

依赖：M1-T01。

验收标准：
- src/lib 可作为统一架构层容器。

#### M2-T02 建立 src/apps/pipecad 目录骨架
目标：创建当前唯一 app 的业务目录模板。

交付物：
- src/apps/CMakeLists.txt
- src/apps/pipecad 目录结构

依赖：M1-T01。

验收标准：
- 未来新增 app 时可直接复用目录模板。

#### M2-T03 迁移 foundation 到 src/lib/base
目标：迁移基础类型、信号、日志、数学工具。

交付物：
- 新路径下的基础头文件与源文件
- 兼容头或别名入口

依赖：M2-T01。

验收标准：
- 原有基础测试可继续构建。

#### M2-T04 迁移 geometry 到 src/lib/platform/occt
目标：迁移 ShapeBuilder、BooleanOps、StepIO、ShapeMesher 等 OCCT 封装。

依赖：M2-T01。

验收标准：
- apps 不再直接耦合 OCCT 头文件。

#### M2-T05 拆分 visualization 与 vtk-visualization
目标：把算法层与 Qt 嵌入层分开，分别进入 lib 和 apps。

依赖：M2-T01、M2-T02。

验收标准：
- lib 侧保留纯渲染/转换能力。
- apps 侧保留视口与 Qt/QML 集成。

#### M2-T06 迁移 app 与 command 到 src/lib/runtime 和 src/lib/framework
目标：迁移 Document、DependencyGraph、ProjectSerializer、CommandStack、CommandRegistry、Application、WorkbenchManager 等架构能力。

依赖：M2-T01。

验收标准：
- 架构核心不再留在旧的 src/app 与 src/command。

#### M2-T07 迁移 model、engine、ui、main 到 src/apps/pipecad
目标：把业务模型、业务引擎、UI 与入口收敛到当前 app。

依赖：M2-T02、M2-T06。

验收标准：
- 当前 app 具备完整独立目录闭包。

### M3 lib 模块化

#### M3-T01 为 base 建立第一批模块接口单元
目标：为 types、signal、log 等提供模块接口单元。

依赖：M2-T03。

验收标准：
- lib/base 至少一组模块可被独立 import。

#### M3-T02 为 platform 建立 façade 模块
目标：收口 OCCT、VSG、VTK 的公开接口面。

依赖：M2-T04、M2-T05、M3-T01。

验收标准：
- apps 不直接依赖第三方头。
- 第三方访问路径统一经过 lib/platform。

#### M3-T03 为 runtime 建立 document、graph、command、task、serialize 模块
目标：让运行时能力具备清晰模块边界。

依赖：M2-T06、M3-T01。

验收标准：
- apps 可通过 import 或公开头访问 lib/runtime。

#### M3-T04 为 framework 建立 app、workbench、scene 模块
目标：完成应用框架模块边界定义。

依赖：M2-T05、M2-T06、M3-T03。

验收标准：
- Application、Workbench、Scene 具备稳定公开接口。

### M4 并发基础设施

#### M4-T01 定义 DocumentSnapshot 契约
目标：形成后台任务只读视图的统一契约。

依赖：M2-T06。

验收标准：
- 后台线程不需要直接访问可变 Document。

#### M4-T02 建立任务队列与线程工作组
目标：提供统一任务调度能力。

依赖：M0-T03、M3-T03。

验收标准：
- 支持任务提交、等待、取消与退出回收。

#### M4-T03 建立结果回投与任务版本控制机制
目标：确保后台任务结果回到主线程时具备时序安全性。

依赖：M4-T02。

验收标准：
- 过期结果可丢弃。
- 主线程回投路径明确。

#### M4-T04 为关键共享状态补齐同步策略
目标：为文档只读访问、dirty 集合消费和场景提交建立同步边界。

依赖：M4-T01、M4-T03。

验收标准：
- 不引入跨线程数据竞争。

### M5 异步重算落地

#### M5-T01 重构 RecomputeEngine 以消费 lib 并发设施
目标：将重算逻辑改为“主线程收集、后台计算、主线程提交”的管线。

依赖：M2-T07、M4 全部任务。

验收标准：
- RecomputeEngine 不直接承担线程管理细节。

#### M5-T02 后台化 ShapeMesher 与批量几何重算
目标：优先处理最明确的性能热点。

依赖：M5-T01。

验收标准：
- 批量几何重算不阻塞 UI 主线程。

#### M5-T03 后台化工程加载后的几何恢复与保存前准备
目标：减少大型工程加载和保存阶段的主线程阻塞。

依赖：M4 全部任务、M5-T01。

验收标准：
- 加载与保存链路具备可控异步阶段。

#### M5-T04 建立并发回归测试
目标：为异步重算、取消、过期结果丢弃和退出回收建立测试覆盖。

依赖：M5-T01 至 M5-T03。

验收标准：
- 至少覆盖批量编辑、撤销重做、工作台切换、应用退出四类场景。

### M6 收口与模板化

#### M6-T01 清理旧目录兼容层
目标：逐步移除旧路径与临时兼容入口。

依赖：M2 至 M5 的主要任务完成。

验收标准：
- src/lib 与 src/apps 成为唯一真实结构。

#### M6-T02 固化当前 app 模板
目标：将 pipecad app 提炼为未来多个 app 的参考模板。

依赖：M2-T07、M6-T01。

验收标准：
- 新增 app 时无需再重新设计 lib 边界。

#### M6-T03 更新文档与开发规范
目标：使文档、构建说明、测试入口与新结构一致。

依赖：M6-T01。

验收标准：
- 开发者仅依赖新结构即可理解和继续开发。

## 8. 关键风险与应对

### 风险 R1：vtk-visualization 拆分难度高
应对：先抽接口，再分离算法与视口集成，不直接做一次性大搬迁。

### 风险 R2：Application 单例渗透范围过大
应对：保留 framework 装配职责，但压缩跨层直接访问，逐步过渡到上下文注入。

### 风险 R3：第三方库头文件继续扩散
应对：优先完成 lib/platform façade，禁止 apps 新增直连依赖。

### 风险 R4：并发引入行为回归
应对：任务基于快照、结果带版本、统一回投主线程，并建立回归测试。

### 风险 R5：迁移期构建系统长期双轨并存
应对：为每个阶段设置明确收口条件，避免兼容层无限期存在。

## 9. 验收标准

本计划执行完成后，应至少满足以下结果：
1. 工程目录收敛为 src/lib 与 src/apps 的长期结构。
2. 工程能产出一个统一架构静态库和至少一个 app 目标。
3. lib 不反向依赖 app。
4. apps 不直接暴露或依赖第三方底层库头文件。
5. lib 内完成第一批模块化落地。
6. 后台任务仅消费快照，不直接修改主线程状态。
7. 编译与测试基线不低于当前已知状态。
8. 文档、构建、测试入口完成同步。

## 10. 推荐执行顺序

建议严格按照以下顺序推进：
1. 先冻结规则与目标命名。
2. 再改 CMake 拓扑。
3. 再做目录骨架与高价值文件迁移。
4. 再推进 lib 模块化。
5. 最后引入并发基础设施与异步重算。

这样做的原因是先稳定结构边界，再引入模块和线程时序变化，可以显著降低调试成本与回滚成本。

## 11. 结论

本计划书将原有高层方案进一步拆解为具备执行顺序、任务编号、依赖关系与验收条件的实施计划。后续如进入编码阶段，可直接按里程碑和任务编号推进，并将完成记录映射回项目任务系统。
